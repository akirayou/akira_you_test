# VR_DoF_Tester 設計仕様

## 1. 目的

高校生向けワークショップで、`3DoF` と `6DoF` の違いをできるだけ安定して体験できる WebXR コンテンツを提供する。

ここで重視するのは「理想的な VR プラットフォームを再現すること」よりも、参加者が短時間で

- 回転だけ反映される状態
- 現実の前進 / 後退 / しゃがみが見え方に反映される状態

の差を体感できること。

## 2. 背景と問題設定

従来は 1 つの A-Frame ページの中で

- `3DoF 単眼`
- `3DoF ゴーグル`
- `6DoF`

を切り替える構成を採っていた。

しかし、`Android + Chrome + 簡易 Cardboard HMD` で `immersive-vr` を使って `6DoF` 比較を成立させるのは不安定で、次の問題がある。

- 端末やブラウザによって positional tracking が利用できない
- WebXR の session / reference space 切り替えが 1 ページ内で複雑になりやすい
- HMD 表示とスマホ側トラッキングの責務が噛み合いにくい
- ワークショップ本番で「動く端末だけ動く」状態になりやすい

そのため、今後は「比較体験の成立」を最優先にして、実装を 2 系統に分ける。

## 3. 設計方針の結論

`VR_DoF_Tester` は今後、次の 2 本立てで設計する。

### A. 安定版: スマホ画面から VR 空間を覗く方式

- こちらを本命とする
- HMD の両眼立体視は使わない
- スマホ画面を窓として、同じ題材空間を `3DoF` と `6DoF` で比較する
- `6DoF` は `immersive-ar` ベースで実位置トラッキングを使う

### B. 実験版: Immersive カメラ位置を HMD 側へ転送する方式

- こちらは研究・検証用プロトタイプとする
- `immersive-ar` などで得た pose を、HMD 表示側のカメラへ転送する
- 動作対象は限定的でよく、本番導線とは切り分ける

## 4. 非目的

今回の設計では、次は目標にしない。

- 1 ページ内で全モードを無理に共存させること
- すべての Android 端末で Cardboard 6DoF 風体験を成立させること
- 専用 VR ヘッドセットと同等の 6DoF 品質をスマホのみで保証すること

## 5. 方式A: 安定版の詳細

## 5.1 体験コンセプト

参加者はスマホを手に持ったまま、画面越しに高所空間を見る。

- `3DoF` では「向きを変えると景色は変わるが、前進しても景色が近づかない」
- `6DoF` では「前進・後退・しゃがみで景色の見え方が変わる」

という差を体感する。

## 5.2 利点

- スマホ単体で成立しやすい
- HMD 固有の UX 問題を避けやすい
- `immersive-ar` による位置追跡を素直に使える
- 実機検証時の切り分けがしやすい

## 5.3 表示モード

安定版では、少なくとも次の 2 モードを用意する。

### `3DoF ビュー`

- 非 HMD
- 端末の向きだけを反映
- 位置は固定
- 高所空間は仮想的に前方へ固定配置

### `6DoF ビュー`

- 非 HMD
- `immersive-ar` セッションを開始
- 床面検知または初期視線方向を基準に、前方へ高所空間を配置
- 実空間での前進 / 後退 / しゃがみを見え方に反映

## 5.4 起動フロー

1. ランチャー画面を表示
2. `3DoF を試す` または `6DoF を試す` を選択
3. `6DoF` 選択時は対応確認を行う
4. `6DoF` では、床面検知または初期配置フェーズを経て空間を固定
5. 比較体験に入る

## 5.5 6DoF 配置ロジック

`6DoF` では次のどちらかで空間を配置する。

- 第一候補:
  床面検知後、床から一定高さの位置に足場と橋を配置する
- 代替候補:
  床面検知が取れない場合、初期カメラ位置と向きを基準に前方固定で配置する

このとき重要なのは「位置移動で見え方が変わること」であり、完全な実世界整合性を最優先にはしない。

## 5.6 成功条件

安定版は、次を満たせば成立とみなす。

- `3DoF` で前進しても見え方がほぼ変わらない
- `6DoF` で前進・後退・しゃがみで見え方が変わる
- 1 分以内に体験を開始できる
- 非対応端末ではその旨が明確に表示される

## 6. 方式B: HMD 転送実験の詳細

## 6.1 目的

簡易 Cardboard HMD でも、実位置トラッキングつきの見え方比較が可能かを検証する。

## 6.2 基本アイデア

- 位置追跡は `immersive-ar` など別経路から取得
- 表示は HMD 用の stereo 表示へ転送
- pose を仮想カメラまたは rig に反映する

## 6.3 位置づけ

- 本番の主要導線にはしない
- 実験ページとして分離する
- 動作端末を限定してよい
- 安定性より検証価値を優先する

## 6.4 実装上の前提

- A-Frame で無理がある場合は Three.js ベースへ寄せてよい
- pose の取得元と適用先はコード上で明確に分ける
- UX より「仕組みが成り立つかどうか」の確認を優先する

## 7. HTML / ファイル分割方針

1 ページで全モードを切り替える構成は避ける。

推奨分割:

- `index.html`
  ランチャー / 説明 / モード選択
- `mode_3dof.html`
  安定版の `3DoF` 体験
- `mode_6dof.html`
  安定版の `6DoF` 体験
- `experiment_hmd_transfer.html`
  HMD 転送実験

必要なら JS も役割別に分離する。

候補:

- `launcher.js`
- `mode_3dof.js`
- `mode_6dof.js`
- `experiment_hmd_transfer.js`
- `shared_scene.js`

## 8. 技術選定方針

## 8.1 共通

- HTML5 / JavaScript
- WebXR Device API
- 大きな外部 3D アセットは使わない
- 教育用途として構造が読みやすいことを優先する

## 8.2 A-Frame を使う箇所

- `3DoF` の単純な視点体験
- 軽量なプリミティブ中心の空間構築
- UI と 3D の素早い組み合わせ

## 8.3 Three.js を使ってよい箇所

- `immersive-ar` の細かな制御
- pose の取得や転送を明示的に扱いたい箇所
- A-Frame の session 切り替えよりも低レベル制御が欲しい箇所

設計上は「A-Frame に統一すること」より「安定した比較体験を実装すること」を優先する。

## 9. 空間演出方針

高所感は必要だが、恐怖演出そのものより「差の知覚」を優先する。

### 必須要素

- 広めの開始足場
- その先に伸びる細い橋
- 下方の地面または都市景観
- 高さや奥行きを知覚しやすいテクスチャや模様

### 調整方針

- 明るさは暗すぎず、奥行きが読みやすいコントラストを保つ
- 地面には模様や格子を入れて移動量が分かりやすいようにする
- `6DoF` では、前進すると橋に近づき、しゃがむと下が覗き込みやすい配置にする

## 10. UI / メッセージ方針

- 対応 / 非対応は必ず明示する
- `6DoF` 非対応時は無言で失敗しない
- 体験前に「何を比べるか」を 1 行で説明する
- 実験ページには、安定版ではないことを明記する

## 11. 実装フェーズ

### Phase 1

- ランチャーを作る
- `3DoF` 単独ページを作る
- `6DoF` 単独ページを作る
- スマホ画面方式で比較体験を成立させる

### Phase 2

- `6DoF` ページの床面検知や初期配置を改善
- 空間演出を比較しやすく調整

### Phase 3

- HMD 転送実験ページを別実装として追加
- 端末限定で検証

## 12. 未決定事項

- `6DoF` の初期配置を hit-test 必須にするか、fallback を許すか
- `3DoF` を A-Frame、`6DoF` を Three.js に分けるか
- ランチャーから各モードへ別 HTML 遷移するか、同一ディレクトリ内の別ページ構成にするか
- 実験ページを公開導線に含めるか

## 13. この spec の役割

この `spec.md` は「現状実装の記録」ではなく、次の大きな作り替えに向けた設計仕様である。

以後の実装判断は、

- 安定した比較体験になるか
- 本番導線と実験導線が分かれているか
- 端末依存による失敗がユーザーに見える形になっているか

を基準に行う。
